URLを叩いてからページが表示されるまでの流れは、以下のステップで構成されています。

1. URL入力
ユーザーがブラウザのアドレスバーにURLを入力し、エンターキーを押す。

2. DNSルックアップ
URL内のドメイン名（例: www.example.com）をIPアドレスに変換する必要があります。これにはDNS（Domain Name System）が使用されます。

ブラウザキャッシュ: ブラウザがまず自分のキャッシュにそのドメインのIPアドレスがないか確認します。
OSキャッシュ: ブラウザキャッシュにない場合、OSのキャッシュを確認します。
ルーターキャッシュ: OSキャッシュにもない場合、ルーターのキャッシュを確認します。
ISPのDNSサーバー: 上記すべてにない場合、ISPのDNSサーバーに問い合わせます。
ルートDNSサーバー: 最終的にルートDNSサーバーに問い合わせ、ドメインのIPアドレスを取得します。
3. TCP接続の確立
IPアドレスが取得できたら、ブラウザはそのIPアドレスに対して接続を開始します。

TCPハンドシェイク: TCP/IPプロトコルを使用して、ブラウザとサーバー間で3ウェイハンドシェイクを行います。
クライアントがサーバーにSYNパケットを送信。
サーバーがSYN-ACKパケットを返信。
クライアントがACKパケットを送信。
4. HTTPSの場合のSSL/TLSハンドシェイク
接続が確立された後、HTTPSを使用する場合、SSL/TLSハンドシェイクを行い、セキュアな通信を確立します。

5. HTTPリクエスト送信
ブラウザはリクエストヘッダーと共にHTTPリクエスト（GETリクエストなど）をサーバーに送信します。

ホスト名: ブラウザはホスト名とリクエストされたリソースを含むURLを送信します。
6. サーバーの処理
サーバーはリクエストを受け取り、処理を開始します。

リソースの検索: リクエストされたリソース（HTMLページ、画像、CSSファイルなど）をサーバー上で探します。
データベースアクセス: 必要に応じてデータベースからデータを取得します。
動的コンテンツ生成: 必要に応じて動的にコンテンツを生成します（例えば、PHPやNode.jsなどを使用して）。
7. HTTPレスポンス送信
サーバーがリクエストを処理し終えたら、HTTPレスポンスをブラウザに送信します。

ステータスコード: リクエストが成功したかどうかを示すステータスコード（例: 200 OK）。
レスポンスヘッダー: レスポンスに関する追加情報を含むヘッダー。
レスポンスボディ: リクエストされたリソース（HTML、画像、CSSなど）。
8. ブラウザのレンダリング
ブラウザがレスポンスを受け取ると、レンダリングプロセスを開始します。

HTML解析: 受け取ったHTMLを解析し、DOMツリーを構築します。
CSS解析: CSSを解析し、スタイルルールを適用してレンダーツリーを構築します。
JavaScriptの実行: スクリプトを順次実行します。
レイアウト: レンダーツリーを基にレイアウトを計算します。
ペイント: 最終的に、各要素を画面に描画します。
9. サブリソースのフェッチ
HTML内に埋め込まれている追加のリソース（画像、CSS、JavaScriptファイルなど）をフェッチします。

パース中のリクエスト: パース中にこれらのリソースをリクエストし、並行してダウンロードします。
追加のレンダリング: これらのリソースがロードされると、再度レンダリングが行われます。
この一連のプロセスにより、ユーザーがURLを入力してからページが表示されるまでの流れが完了します。

## DNSとは
DNS（Domain Name System）は、インターネット上で使用されるドメイン名をIPアドレスに変換するシステムです。
DNSの基本的な仕組み
ドメイン名の構造

ルートドメイン: 最上位に位置するドメインで、ピリオド「.」で示される。通常は省略される。
トップレベルドメイン（TLD）: ルートドメインの下に位置し、例えば.com、.org、.netなど。
セカンドレベルドメイン: TLDの直下に位置し、例えばexample.comの「example」部分。
サブドメイン: セカンドレベルドメインの下に位置し、例えばwww.example.comの「www」部分。
DNSの役割

ドメイン名をIPアドレスに変換する（例: www.example.com → 192.0.2.1）。
電子メールの配送先を指定するためのMXレコードなど、他のリソースレコードを提供する。
DNSの動作プロセス
ドメイン名の入力
ユーザーがブラウザのアドレスバーにドメイン名（URL）を入力し、エンターキーを押します。

DNSキャッシュの確認

ブラウザキャッシュ: ブラウザはキャッシュにそのドメインのIPアドレスがないか確認します。
OSキャッシュ: ブラウザキャッシュにない場合、OSのキャッシュを確認します。
ルーターキャッシュ: OSキャッシュにもない場合、ルーターのキャッシュを確認します。
ISPのDNSキャッシュ: 上記すべてにない場合、ISPのDNSサーバーに問い合わせます。
再帰的なDNSクエリ
キャッシュにない場合、再帰的なDNSクエリが行われます。

ルートDNSサーバーへの問い合わせ: ドメイン名の最後の部分（TLD）を解決するためにルートDNSサーバーに問い合わせます。
TLD DNSサーバーへの問い合わせ: ルートDNSサーバーはTLD DNSサーバー（例: .com DNSサーバー）のアドレスを返します。
権威DNSサーバーへの問い合わせ: TLD DNSサーバーは、ドメイン名の権威DNSサーバー（例: example.comのDNSサーバー）のアドレスを返します。
権威DNSサーバーの応答
権威DNSサーバーがドメイン名に対応するIPアドレスを返します（例: 192.0.2.1）。

DNSレスポンスのキャッシュ
取得したIPアドレスは、ブラウザやOSのキャッシュに保存され、今後同じドメイン名へのアクセス時に再度問い合わせを行わずに済むようにします。

DNSレコードの種類
Aレコード（Address Record）: ドメイン名をIPv4アドレスにマッピングします。
AAAAレコード: ドメイン名をIPv6アドレスにマッピングします。
CNAMEレコード（Canonical Name Record）: 別のドメイン名にエイリアスします。主にサブドメインのために使用されます。
MXレコード（Mail Exchange Record）: メールサーバーのIPアドレスを指定します。
NSレコード（Name Server Record）: ドメインの権威DNSサーバーを指定します。
TXTレコード（Text Record）: 任意のテキスト情報を保持します。SPFレコードやDKIMなどの認証に使用されます。
SRVレコード（Service Record）: 特定のサービスの場所を指定します。
DNSの重要な特徴
分散性: DNSは分散型システムであり、全世界の多数のサーバーが協力して動作しています。
キャッシング: DNSクエリの結果はキャッシュされ、後続のクエリが高速化されます。
再帰的および反復的クエリ: 再帰的クエリはクライアントからの要求をDNSサーバーが他のDNSサーバーに問い合わせて解決するもので、反復的クエリはクライアント自身が各DNSサーバーに問い合わせを行うものです。
この仕組みを通じて、DNSはユーザーが覚えやすいドメイン名を使用してインターネット上のリソースにアクセスできるようにします。


## HTTPステータスコード
|CODE|説明|
|---|---|
|1xx| Informational|
|100| Continue: サーバーがリクエストを受け取り、クライアントがリクエストを続行することを指示。|
|101 |Switching Protocols: クライアントの要求に従ってプロトコルを切り替えた。|
|2xx:| Success|
|200 |OK: リクエストが成功し、レスポンスと共に返された。|
|201 |Created: リクエストが成功し、リソースが作成された。|
|202 |Accepted: リクエストが受け付けられたが、処理は完了していない。|
|204 |No Content: リクエストは成功したが、返すコンテンツがない。|
|3xx:| Redirection|
|300 |Multiple Choices: リクエストに複数の応答があり、どれを選ぶか決定する必要がある。|
|301 |Moved Permanently: リクエストされたリソースが恒久的に移動された。|
|302 |Found: リクエストされたリソースが一時的に移動された。|
|304 |Not Modified: リソースは変更されていないので、再送信しない。|
|4xx:| Client Error|
|400 |Bad Request: リクエストが不正である。|
|401 |Unauthorized: 認証が必要である。|
|403 |Forbidden: リクエストが拒否された。|
|404 |Not Found: リクエストされたリソースが見つからなかった。|
|405 |Method Not Allowed: 指定されたメソッドは許可されていない。|
|409 |Conflict: リクエストがリソースの現在の状態と矛盾している。|
|429 |Too Many Requests: リクエストが多すぎる。|
|5xx:| Server Error|
|500 |Internal Server Error: サーバー内部のエラー。|
|501| Not Implemented: リクエストを処理する機能がサーバーにない。|
|502| Bad Gateway: サーバーがゲートウェイまたはプロキシとして機能しており、不正なレスポンスを受け取った。|
|503| Service Unavailable: サービスが利用できない（過負荷やメンテナンス中など）。|
|504| Gateway Timeout: ゲートウェイまたはプロキシが、上流サーバーからタイムアウト応答を受け取った。|


これらのコードは、クライアントとサーバー間の通信状況やエラーの種類を示すために使用されます。
